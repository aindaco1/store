# Store v1.0.8 Release Candidate and Closure Plan

Date opened: 2026-07-13

Release status: **candidate — not tagged or deployed**

The Storefront package, Worker package, canonical platform version, request identifiers, changelog, roadmap, and Pool adaptation record identify this work as `1.0.8`. The published `v1.0.7` tag remains immutable. Do not add a `v1.0.8` release date or tag until every required closure row below has evidence or an explicitly approved non-applicable decision.

## Scope

- Root repository operating guidance and docs-as-code alignment
- DRY optional add-on variant pricing
- Repository-authoritative image, video, and audio management and optimization evidence
- Stripe request, webhook, journal, and read-only reconciliation hardening
- Durable Resend outbox, delivery evidence, and suppression
- Production quality/admin follow-through
- Evidence-based Cloudflare Workers Cache rollout or no-rollout decision
- Protected backup, restore, and disaster-recovery operational proof

Tax expansion and the guided setup TUI remain outside `v1.0.8`.

### 3. Cloudflare Workers Cache decision

Owner: Store performance operator

Actions:

1. **Implemented in the release candidate:** “no optional cache route enabled” is an explicit successful `not_applicable`/`no_enabled_candidates` state only after the authenticated request succeeds and sanitization, three-read budget, route matching, consistent `DISABLED`, unchanged-repeat, and zero-repeat-read assertions pass. `WORKERS_CACHE_EVIDENCE_ROUTE` selects the sequential candidate. Malformed data, leaks, unsafe or mismatched routes, excess requests, mixed enabled/disabled evidence, or an enabled candidate that misses its gate fail closed.
2. Add focused tests for disabled/no-candidate, deployment warmup, insufficient traffic, enabled-pass, enabled-fail, and sanitized artifact behavior.
3. After the `1.0.8` deployment is stable for at least four hours, capture deployment-scoped aggregate evidence.
4. Keep Orders disabled. Benchmark Analytics with at least 30 disabled and 30 enabled samples, mutation/purge freshness, zero warm/no-change order-data KV list/get work, and the existing 40-percent p95 improvement threshold.
5. If Analytics passes, enable only Analytics, observe hit ratio/stale/fallback behavior, and prove the kill switch/rollback. If it fails, disable it and record a no-rollout decision.
6. Evaluate order-derived inventory and then download readiness with the same gate only after the prior candidate closes. Do not enable multiple new routes in one experiment.

Acceptance:

- The scheduled job truthfully passes when caching is intentionally inapplicable and fails when an enabled route violates policy.
- Each evaluated route has sanitized 30-sample evidence and an explicit enable/disable decision. A measured no-rollout is a valid release outcome.
- `cross_version_cache` remains false unless separately proven safe and beneficial.

Rollback:

- Disable the route switch, deploy the reviewed config, purge the affected policy/tag, verify uncached freshness, and retain sanitized evidence under the incident procedure.

### 4. Protected recovery proof

Owner: Store recovery operator and required protected-environment reviewer

Actions:

1. **Completed on 2026-07-13:** following Pool's pattern, provision a private Standard-class R2 archive in a Cloudflare account separate from production. Credentials are Object Read & Write for the Store bucket only, the destination uses the `store/` prefix, no public URL or custom domain is enabled, and the prefix has a 400-day immutable lock.
2. **Implemented in the release candidate:** before customer-data capture, require distinct production/archive account IDs, a matching Cloudflare R2 endpoint, region `auto`, a Store-only prefix, and exactly 400 lock days; then upload a non-sensitive canary, verify byte-identical readback, require deletion rejection, and verify post-rejection readback. After the target exists, retain the resulting sanitized proof.
3. Create a fresh one-time super-admin token immediately before capture. Run a complete encrypted `v1.0.8` snapshot including admin CSV/readiness/download exports and complete KV/R2 enumeration.
4. Record snapshot duration plus aggregate KV/R2/admin-export operations and compare them with the approved four-hour active-sales RPO and documented RTO.
5. Download/copy the encrypted archive to a second isolated device/location; verify manifest checksums and decryptability there; remove plaintext temporary files.
6. Run the aggregate captured-order comparison with the provisioned restricted live-mode Payment Intents read-only Stripe key. Fail before provider access on mode mismatch and execute no Stripe write.
7. Enable the protected quarterly job only after its environment, reviewer, age identity, fresh token, restricted Stripe key, off-account destination, and preview KV/R2 resources are verified.
8. Execute preview-only restore, value/object readback, exact-snapshot cleanup, and zero-residual verification. Exercise best-effort cleanup for a controlled failure fixture without touching production.
9. Run recovery readiness again with the current encrypted receipt and record no warning for snapshot age.

Acceptance:

- One immutable off-account encrypted release snapshot exists and is decryptable from a second isolated location.
- The protected drill includes admin exports and live read-only Stripe comparison, restores only approved preview state, excludes quarantine/derived state as designed, and leaves zero snapshot-owned preview data.
- Duration/operation evidence supports the approved RPO/RTO or records an owner and dated corrective action.

Rollback/safety:

- Keep `RECOVERY_DRILL_ENABLED=false` until every prerequisite is present. The automated workflow never targets production restore. Production recovery remains a manual incident process with maintenance, paused Stripe webhooks, inventory review, a distinct verified pre-restore snapshot, maker/checker controls where required, and exact acknowledgement.
