# Store Email System

Store sends transactional email through Resend from the Worker. Templates live in `worker/src/email.js`, copy lives in `_data/i18n/*.yml`, and order/reminder scheduling lives in `worker/src/index.js`.

Since `v1.0.4`, Store order receipts live fully in this path: Stripe PaymentIntents do not request Stripe receipt emails, customers receive Store-owned order confirmations, and super admins receive lightweight order notifications.

## Senders

- Order and lookup emails use `ORDERS_EMAIL_FROM`.
- Admin access, abandoned checkout, and event reminder emails use `UPDATES_EMAIL_FROM` with `ORDERS_EMAIL_FROM` as fallback.
- Replies default to `SUPPORT_EMAIL`.

## Email Types

### Store Order Admin Notification

Sent to effective `super_admin` users after a new Store order is confirmed.

- Free RSVP/free orders notify admins after inventory is committed.
- Paid orders notify admins after Stripe confirms the PaymentIntent through webhook settlement.
- Limited Store admins do not receive this notification automatically.
- Notifications use the existing Store order email renderer without ticket/calendar/QR attachments, keeping the admin message lightweight and transactional.
- The admin CTA includes a short-lived one-time `admin_login` link for the Orders tab so super admins can open the order without requesting a second sign-in email.
- Order-notification admin links expire after 5 minutes and create a 30-minute admin session after successful exchange. They are bearer links, so forwarding an unused notification can still delegate access until the link is consumed or expires.

### Admin Access

Sent when an admin user is added from the dashboard and notification delivery is enabled for that action.

The email tells the user what access was added and points them to the admin sign-in page. It does not include a password.

### Admin Magic Link

Sent when an admin requests sign-in. The link is short-lived and generated by the admin auth flow.

### Order Confirmation

Sent once to the customer after a Store order is confirmed:

- Free RSVP/free orders send after inventory is committed.
- Paid orders send after Stripe confirms the PaymentIntent through webhook settlement.
- Historical Snipcart imports do not send customer emails.
- Stripe is not asked to send receipts for Store PaymentIntents; Store owns the customer order email through Resend so receipt copy, localization, and fulfillment links stay in one path.

Order confirmations include order totals, item rows, shipping address when applicable, and a link to `/order-success/?orderToken=...`.

For ticket or RSVP items, confirmations attach only low-risk calendar artifacts when available:

- `.ics` calendar invite when the product has Calendar enabled and a valid start time.

Tickets and check-in QR codes stay on the authenticated order page instead of being attached as SVG files. SVG attachments are avoided for deliverability and usability: many clients treat SVGs as risky active content, preview them inconsistently, or hide them behind attachment warnings.

### Order Lookup

Sent when a customer requests order lookup by email and matching orders exist. The email contains a one-time link to `/orders/` backed by `store-order-lookup:*` KV records. Tokens currently expire after 15 minutes.

Order lookup signs tokens with `STORE_ORDER_LOOKUP_SECRET` when configured, then falls back through Store fulfillment/download and magic-link secrets.

### Abandoned Checkout

Sent only when the customer asked for a checkout reminder. The default send delay is 6 hours after checkout starts. The email contains:

- A resume checkout link.
- A one-click unsubscribe link.
- List-Unsubscribe headers.

Queue, sent, suppression, and health records live in `STORE_STATE` KV with `abandoned-cart:*` keys.

Use `ABANDONED_CART_TOKEN_SECRET` for dedicated reminder resume/unsubscribe signing when possible. If it is not configured, Store falls back to checkout intent or magic-link signing secrets.

### Event Reminders

Sent for confirmed ticket or RSVP order items with a valid start time. Reminder records are queued when the order is confirmed.

Reminder cadence:

- 1 week before the event.
- 1 day before the event.
- 6 hours before the event.
- 1 hour before the event.

Each reminder includes the event name, time, venue/address when configured, a link to the order page, and the same event attachments used by order confirmations.

Queue and sent records live in `STORE_STATE` KV:

- `store-event-reminder:*`
- `store-event-reminder-sent:*`
- `store-event-reminder-queue:v1`

The cron handler processes due reminders and retries failures with bounded backoff.

Cron status and recent errors are visible in **Settings -> Store readiness** and through the authenticated cron/observability endpoints.

## Calendar And QR Behavior

Calendar files are generated from structured product event fields:

- `starts_at`
- `ends_at`
- `venue`
- `address`
- `ics`

The order page calendar download uses `METHOD:PUBLISH`. Email attachments use `METHOD:REQUEST` so email clients treat them as invites.

Check-in QR codes point to signed Worker check-in URLs. Email QR tokens are signed to remain valid through the event window plus one day.

## Copy And Localization

Human-facing copy should stay short, direct, and localized through `_data/i18n/en.yml` and `_data/i18n/es.yml`.

Templates fall back to English strings in `worker/src/email.js` when a catalog key is missing.

## Testing

Unit coverage includes:

- Order confirmation email payloads and attachment passthrough.
- Super-admin order notification payloads.
- Event reminder email payloads.
- Event order attachment generation.
- Reminder queue creation and due reminder delivery.
- Order lookup delivery.

Run the focused suite with:

```sh
npm run test:unit -- tests/unit/email-store.test.ts tests/unit/event-reminders.test.ts tests/unit/order-lookup-email.test.ts
```

For release payment evidence, run the target Worker with `STORE_EMAIL_DRY_RUN=true` or `RESEND_EMAIL_DRY_RUN=true`. The Worker records email delivery markers without calling Resend, and `npm run release:payment-smoke -- --direct-webhook` verifies customer/admin order emails would render for the local signed-webhook matrix.
