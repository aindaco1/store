# Store Email System

Store sends transactional email through Resend from the Worker. Templates live in `worker/src/email.js`, copy lives in `_data/i18n/*.yml`, and order/reminder scheduling lives in `worker/src/index.js`.

## Senders

- Order and lookup emails use `ORDERS_EMAIL_FROM`.
- Admin access, abandoned checkout, and event reminder emails use `UPDATES_EMAIL_FROM` with `ORDERS_EMAIL_FROM` as fallback.
- Replies default to `SUPPORT_EMAIL`.

## Email Types

### Admin Access

Sent when an admin user is added from the dashboard and notification delivery is enabled for that action.

The email tells the user what access was added and points them to the admin sign-in page. It does not include a password.

### Admin Magic Link

Sent when an admin requests sign-in. The link is short-lived and generated by the admin auth flow.

### Order Confirmation

Sent once after a Store order is confirmed:

- Free RSVP/free orders send after inventory is committed.
- Paid orders send after Stripe confirms the PaymentIntent through webhook settlement.
- Historical Snipcart imports do not send customer emails.

Order confirmations include order totals, item rows, shipping address when applicable, and a link to `/order-success/?orderToken=...`.

For ticket or RSVP items, confirmations attach event artifacts when available:

- `.ics` calendar invite when the product has Calendar enabled and a valid start time.
- Ticket/RSVP SVG with embedded check-in QR.
- Standalone check-in QR SVG.

### Order Lookup

Sent when a customer requests order lookup by email and matching orders exist. The email contains a one-time link to `/orders/` that expires quickly.

### Abandoned Checkout

Sent only when the customer asked for a checkout reminder. The email contains:

- A resume checkout link.
- A one-click unsubscribe link.
- List-Unsubscribe headers.

Queue, sent, suppression, and health records live in `STORE_STATE` KV with `abandoned-cart:*` keys.

### Event Reminders

Sent for confirmed ticket or RSVP order items with a valid start time. Reminder records are queued when the order is confirmed.

Reminder cadence:

- 1 week before the event.
- 1 day before the event.
- 6 hours before the event.
- 1 hour before the event.

Each reminder includes the event name, time, venue/address when configured, a link to the order page, and the same event attachments used by order confirmations.

Queue and sent records live in `STORE_STATE` KV:

- `store-event-reminder:*`
- `store-event-reminder-sent:*`
- `store-event-reminder-queue:v1`

The cron handler processes due reminders and retries failures with bounded backoff.

## Calendar And QR Behavior

Calendar files are generated from structured product event fields:

- `starts_at`
- `ends_at`
- `venue`
- `address`
- `ics`

The order page calendar download uses `METHOD:PUBLISH`. Email attachments use `METHOD:REQUEST` so email clients treat them as invites.

Check-in QR codes point to signed Worker check-in URLs. Email QR tokens are signed to remain valid through the event window plus one day.

## Copy And Localization

Human-facing copy should stay short, direct, and localized through `_data/i18n/en.yml` and `_data/i18n/es.yml`.

Templates fall back to English strings in `worker/src/email.js` when a catalog key is missing.

## Testing

Unit coverage includes:

- Order confirmation email payloads and attachment passthrough.
- Event reminder email payloads.
- Event order attachment generation.
- Reminder queue creation and due reminder delivery.
- Order lookup delivery.

Run the focused suite with:

```sh
npm run test:unit -- tests/unit/email-store.test.ts tests/unit/event-reminders.test.ts tests/unit/order-lookup-email.test.ts
```
