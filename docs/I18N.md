# Store Localization

Store ships with English as the default locale plus Spanish UI strings and language-prefixed product presentation pages.

## Release v1.0.4 Audit

- New Store order, super-admin order notification, admin magic-link, event reminder, and fulfillment copy is mirrored in English and Spanish.
- Public home, Terms, Orders, and Order Success shells now have Spanish routes. Product names, descriptions, and creator-authored product body copy remain canonical unless a product explicitly provides localized overrides.
- The Spanish Terms page is machine translated for the launch branch and includes a notice that English controls until reviewed.
- `npm run test:i18n` compares supported locale files against English and is part of the release check path.
- Store-owned order emails use `_data/i18n/{lang}.yml` through the shared email translator instead of hardcoded Spanish branches.
- Product names/descriptions remain creator-authored catalog content and are not automatically translated.

## Configuration

Locales are declared in `_config.yml`:

```yaml
i18n:
  default_lang: en
  supported_langs:
    - en
    - es
  language_labels:
    en: English
    es: Español
  product_path_prefixes:
    en: /products/
    es: /es/products/
  pages:
    home:
      en: /
      es: /es/
    terms:
      en: /terms/
      es: /es/terms/
    orders:
      en: /orders/
      es: /es/orders/
    order_success:
      en: /order-success/
      es: /es/order-success/
    admin:
      en: /admin/
      es: /es/admin/
```

Add a language by:

1. Adding the locale to `i18n.supported_langs`.
2. Adding `i18n.language_labels`.
3. Adding route entries under `i18n.pages`.
4. Adding or updating `_data/i18n/{lang}.yml`.
5. Adding source pages for localized public routes.

## Shared Includes

Core localization helpers:

- `_includes/t.html` for translation lookup
- `_includes/localized-url.html` for route lookup
- `_includes/language-switcher.html` for footer/admin route switching
- `_includes/localized-date.html`
- `_includes/localized-datetime.html`

Public templates should pass `lang=current_lang` when rendering translated UI strings.

## Store Pages

Current canonical public pages:

- `/`
- `/es/`
- `/terms/`
- `/es/terms/`
- `/products/:slug/`
- `/:lang/products/:slug/` for generated localized product presentation pages
- `/orders/`
- `/es/orders/`
- `/order-success/`
- `/es/order-success/`
- `/admin/`
- `/es/admin/`

Product descriptions, product names, and product bodies are creator/operator-authored catalog content. They are not automatically translated.

## Product Pages

Store uses one canonical product file in `_products/` and generated presentation routes for non-default locales. Commerce fields stay canonical:

- `identifier`
- `sku`
- `price`
- `variants`
- inventory
- fulfillment metadata
- shipping and tax metadata

The generated pages live outside the `products` collection, use `store_product: true`, and render the same product card with localized UI chrome. This avoids duplicate sellable catalog entries while still allowing SEO alternates and a language switcher.

Optional product-authored translations live in the canonical product file:

```yaml
localized:
  es:
    slug: fronteras-camiseta
    name: Camiseta Fronteras
    description: Camiseta oficial del festival.
    body: |
      Camiseta oficial del Fronteras Micro-film Festival.
```

If a locale entry is missing, Store still generates the localized URL and falls back to the canonical product name, description, and body. Do not create separate sellable product entries per locale without a migration plan, because product URLs affect checkout metadata, SEO, email links, and admin publishing.

## Admin

Admin UI copy lives in `_data/i18n/en.yml` and any added locale files under the `admin` and `runtime` sections.

Admin language switching must preserve safe view state and strip sensitive login tokens from alternate-language links.

Generated field labels should use deterministic keys so additions are straightforward:

- `settings_field_*`
- `settings_readonly_*`
- `store_product_*`
- `store_coupon_*`
- `store_order_*`
- `store_download_*`
- `store_inventory_*`
- `store_marketing_*`

## Runtime Copy

Runtime browser copy should come from `_includes/runtime-messages-json.html`, then be read by Store JavaScript modules.

Store runtime keys should cover:

- cart labels and errors
- checkout labels and validation copy
- shipping/tax helper text
- Order Success states
- Order lookup states
- coupon labels/errors
- admin dashboard status copy
- download/ticket/RSVP fulfillment labels
- marketing/reminder labels

Avoid hardcoding new user-facing strings in JavaScript when the surrounding module already reads runtime messages.

## Completeness Check

Run this before adding or changing public/admin runtime copy:

```bash
npm run test:i18n
```

The check reads `_config.yml` supported locales and compares every non-default locale against English keys in `_data/i18n/en.yml`. Missing or extra keys fail the check so runtime and email copy do not silently fall back because of catalog drift.

## Email Templates

Store email templates load localized copy through `_data/i18n/{lang}.yml` and `assets/i18n.json`.

Covered transactional templates:

- admin magic links and admin access notifications
- customer order confirmations for physical, digital, ticket, and RSVP items
- super-admin order notifications
- customer order lookup links
- abandoned checkout reminders
- event reminders with ticket/RSVP artifacts

## SEO

SEO includes use:

- `page.lang`
- `page.translation_key`
- localized canonical/alternate routes when available
- `availableLanguage` in JSON-LD
- product-specific metadata on product pages

For product pages, Store emits product metadata from `_products`, adds `localized_paths`, and emits canonical/alternate links for generated language routes.

## Checklist For A New Locale

1. Add locale config in `_config.yml`.
2. Create `_data/i18n/{lang}.yml`.
3. Add localized source pages for public page shells such as home, terms, orders, order success, and admin.
4. Confirm `/admin/` and localized admin route are `noindex`.
5. Run `bundle exec jekyll build --quiet`.
6. Run `npx vitest run tests/unit/seo-layouts.test.ts`.
7. Run the default Playwright suite at desktop and mobile breakpoints.
8. Manually verify cart, checkout, Order Success, and admin status strings.

## Out Of Scope For Now

- automatic translation of product names/descriptions
- locale-specific tax/shipping policy
